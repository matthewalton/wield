# Strengthen findings — push slice, 2026-07-14

A `strengthen` pass on `src/push` (Baton #123, Wield), run by an agent end to
end. The slice was green and badly defended: every one of its 143 mutants was
no-coverage, because the pure modules (`exposition.ts`, `remote-write.ts`)
were exercised only through `cli.test.ts`'s spawn-the-CLI tests, and mutant
activation never crosses the child-process boundary (ADR-0006). This run is a
worked example of every routing exit in the skill: machine path, human path,
equivalent mutant — and the trap, declined.

## Finding 1 — a real wire-format bug (machine path)

The new `remote-write.test.ts` asserts byte-exact expected buffers hand-derived
from the prompb and snappy block-format specs (the stance decisions/0002 set:
conformance against the spec, not a mirror of the encoder). The very first run
of the snappy framing-tier test failed — not the test, the encoder:

- `snappyCompress` allocated a **5-byte literal tag for payloads ≥ 65537
  bytes**, but tag 62 is the 3-byte-length form — 4 bytes total. The stray
  `0x00` landed between the length bytes and the data, corrupting the stream.
- Invisible to the spawn tests: no CLI test pushes a >64 KB payload, and the
  independent decoder that would have rejected it never saw one that size.

One-byte fix (`Buffer.alloc(5)` → `Buffer.alloc(4)`), now pinned by a test
that brackets every literal-tag length tier (1, 60, 61, 127, 128, 256, 257,
65536, 65537 bytes).

## Finding 2 — the spec was silent about its own drift alarm (human path)

Every remaining `exposition.ts` survivor sat in the parser's rejection paths:
unterminated label value, series without labels, malformed label, unreadable
sample value. No assertion following from PUSH-1 can reach them — PUSH-1 is a
promise about well-formed renderer output. But the behaviour matters: the
parser is the drift alarm for the wire-compatibility contract (the push
delivers exactly what the rendering prints, decisions/0001), and the mutants
turned "fail loudly" into "deliver mis-read series".

So the spec owed a criterion. Drafted this run, lint-clean:

> **[PUSH-15]** Exposition text that cannot be read back into series is
> rejected with an error saying what was unreadable and naming the offending
> line.

Five killing tests followed from its statement. Rising oracle strength and a
sharpening spec were the same motion: the survivors exposed a promise the spec
had never written down.

## Finding 3 — four equivalent mutants, argued from semantics

Annotated where they live, each with its argument in the code:

- Two `exposition.ts` mutants only alter a value that the unterminated-value
  throw then discards — no observable difference on any input.
- `Buffer.from(name, "utf8")` → `Buffer.from(name, "")`: Node treats a falsy
  encoding as utf8; the mutant is the same encoder (verified empirically).
- `a < b` vs `a <= b` in the label-name comparator: label names within a
  series are distinct (the renderer dedupes; duplicates are protocol-invalid),
  so equality never occurs.

Two code simplifications fell out of arguing them: the sort comparator's
`a > b ? 1 : 0` arm was unreachable-in-effect under V8's sign-only sorts, so
the comparator became `a < b ? -1 : 1` with the distinct-names argument as a
comment; and a `.trim()` before `Number(...)` was deleted outright because
StringToNumber already ignores surrounding whitespace — dead code beats an
annotation.

## The trap, declined

PUSH-1's five remaining survivors live in the **scanner's** `prom.ts` (HELP/
TYPE header strings, the sanitize-warning conditional) — the push round-trip
tests cover them, so they count against PUSH-1. Asserting HELP text from a
push test would kill them and turn the bar green, and it would be
test-fitting: PUSH-1 says nothing about comment lines. They are routed to the
scanner slice's own strengthen pass (Wield Baton #127), and because a kill
credits every covering criterion (Speccle ADR-0011), fixing them there lifts
PUSH-1 to 100% for free.
